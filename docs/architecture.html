How It Works

Architecture

One engine, many vaults. No database, no embeddings, no server. Just files and a capable model.

Three Layers
The idea is related in spirit to Vannevar Bush's Memex (1945) — a personal knowledge store with associative trails between documents. The part he couldn't solve was who does the maintenance. The LLM handles that. — Andrej Karpathy

Layer 1: Raw Sources

Your curated collection of source documents. Articles, papers, PDFs, videos, tweets, gists. Immutable — the LLM reads from them but never modifies. This is your source of truth.

vaults/name/raw/

Layer 2: The Wiki

LLM-generated markdown files. Summaries, entity pages, concept pages, comparisons. The LLM owns this layer entirely. It creates pages, maintains cross-references, keeps everything consistent.

vaults/name/wiki/

Layer 3: The Schema

A conventions document that tells the LLM how the wiki is structured, what workflows to follow, what frontmatter to use. You and the LLM co-evolve this over time.

vaults/name/CLAUDE.md
Engine & Vaults

One engine, many vaults

Claude Code is the engine — 25 skills in .claude/skills/. Vaults are dumb folders of markdown, each its own git repo. The engine operates on any vault. Vaults contain no logic.

engine (this repo)
llm-wiki/ ├── .claude/skills/ <- 25 skills ├── .private/ <- your private skills ├── vaults/ <- gitignored ├── docs/ <- this site └── CLAUDE.md
vault (separate repo)
vaults/my-research/ ├── raw/ <- sources ├── wiki/ │ ├── index.md │ ├── sources/ │ ├── entities/ │ ├── concepts/ │ └── comparisons/ ├── log.md └── CLAUDE.md
Wiki Pages

5 page types, consistent frontmatter

Every page has YAML frontmatter with title, dates, page-type, domain tags, source links, and related pages.

Source Note

Summary of one ingested source. Full provenance: URL, author, date, raw file path, source type.

wiki/sources/

Entity

A person, organization, tool, or framework. Has entity-type and aliases fields.

wiki/entities/

Concept

An idea, pattern, or technique. Definition, how it works, examples, related concepts.

wiki/concepts/

Comparison

Synthesis comparing two or more approaches. Table, analysis, verdict.

wiki/comparisons/

Summary

High-level overview synthesizing across sources. Key themes, open questions.

wiki/
Navigation

index.md + log.md

index.md

Content-oriented catalog. Tables grouped by type (sources, entities, concepts, comparisons). Claude reads this first when querying. Works well up to ~200 pages.

## Sources | Page | Summary | Domain | |------|---------|--------| | [[article]] | Key ideas... | ai |

log.md

Chronological activity record. Parseable with grep "^## \[" log.md. Ingests, queries, lint passes, promotions — everything timestamped.

## [2026-04-12] ingest | Article - Source type: article - Pages created: [[source]], [[entity]] ---
Obsidian Integration

Direct file access, no MCP

Claude Code reads and writes markdown files directly. Obsidian watches the same directory and renders changes in real time. No middleware, no MCP server, no plugins required.

Claude Code --writes--> filesystem --renders--> Obsidian

MCP servers exist but add complexity for zero functional gain — Claude Code already reads files. Add qmd for search when your wiki exceeds ~200 pages.

Private Overlay

Your skills stay private

The .private/ directory is gitignored. Put employer-specific or personal skills there. Initialize it as its own git repo. Pull engine updates without touching your private stuff. 

# Pull engine updates anytime:
$ git remote add upstream https://github.com/RonanCodes/llm-wiki.git
$ git pull upstream main
# Your .private/ and vaults/ are untouched
