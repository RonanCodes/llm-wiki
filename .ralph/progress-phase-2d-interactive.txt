## 2026-04-18 12:08 - US-001: Implement generate-quiz handler

- Created `.claude/skills/generate-quiz/SKILL.md` — pure text pipeline (no runtime deps), awk-based template fill to avoid sed escaping pain on JSON braces/quotes.
- Shipped `templates/quiz.html` — single-file, Observatory palette, vanilla JS. 270 lines including CSS. MCQ + short-answer. Per-question explanation reveal. Score panel. No bundler, no framework.
- Questions schema supports `mcq` (options + correct index) and `short` (correct_keywords for lenient client-side grading). Each question carries a `source` wiki page path — critical for Phase 2E fidelity testing.
- `.questions.json` kept alongside `.html` (same pattern as `.script.md`, `.outline.md`, `scenes.json`).
- LocalStorage persistence intentionally omitted — "one-liner to add when anyone asks".
- **Learnings for future iterations:**
  - awk beats sed for template fill when placeholder content is multi-line JSON — sed's newline-in-replacement pain isn't worth the 2 fewer characters.
  - Client-side grading via keyword match is "good enough for self-study, not for assessment". Documented that limitation explicitly rather than trying to solve it — the right tool for strict grading is a different artifact type.
  - Single-file HTML with inlined CSS + JS + JSON is the right default for shareable study artifacts: email-able, offline, no hosting. Same spirit as markmap's output.
  - Per-question `source` field is the fidelity-test hook. When Phase 2E's `verify-artifact` runs, it'll re-check each question's claims against the cited wiki page. Designing for that loop from the start.

## 2026-04-18 12:42 - US-002: Implement generate-flashcards handler

- Created `.claude/skills/generate-flashcards/SKILL.md` — 8-step pipeline: dep check → resolve pages → cards.csv (LLM) → genanki Python build → keep csv alongside → sidecar → commit → report.
- Lazy-install chain: `python3 -c "import genanki"` probe, then `pipx install genanki` if available, else `pip install --user genanki`. Same pattern as pandoc/marp/yt-dlp.
- **Stable deterministic IDs** — `deck_id = sha256(deck_name)[:8]` and `model_id = sha256(b"llm-wiki-basic-v1")[:8]`. This is the critical UX decision: Anki updates cards in-place on re-import when both IDs match. Random IDs would create duplicate decks on every render — silently painful. Model_id version-string (`v1`) leaves room for a future cloze model without breaking existing cards.
- Deck naming: `llm-wiki::<vault>::<topic>`. The `::` separator is Anki's hierarchical folder convention — keeps the sidebar tidy when a user generates decks across multiple vaults.
- 4-column CSV schema: `front, back, source, tags`. RFC 4180 quoting (LLM instructed to double embedded `"`). `source` is mandatory — rendered in the card footer at review time, and is the Phase 2E fidelity-check hook.
- Python helper is heredoc'd inline rather than checked in as `build_deck.py`. ~30 lines — the duplication cost of regenerating it at render time is negligible and it keeps the skill self-contained (one file, no separate assets beyond templates).
- **Learnings for future iterations:**
  - CSV beats JSON for flashcard sidecars. Anki's own import path is CSV-native, and `awk`/`cut` keep the diff readable when you re-generate. JSON's escaping would win for complex fields but flashcards are 99% plain text.
  - Stable deterministic deck IDs are the single biggest UX lever for any "re-generate" artifact that has a native app around it (Anki, Calibre, etc.). Always derive from a human-stable string, not a timestamp.
  - Hierarchical deck names (`::` separator) are free win in Anki — users get a folder tree without any extra config.
  - Model-ID versioning (`v1`) costs nothing now and buys room for future models (cloze, image-occlusion) without clobbering users' review history.
  - Anki `.apkg` import doesn't require a local Anki install to generate the file — it's just SQLite under the hood. That makes this handler pure Python + no daemon, same spirit as markmap's self-contained HTML.

## 2026-04-18 13:05 - US-003: Implement generate-app handler (scaffolder)

- Created `.claude/skills/generate-app/SKILL.md` — 9-step pipeline: dep check → resolve → LLM picks template → LLM writes data.json → `cp -r template/ → app-dir/` + write data.json → README → sidecar → commit → report.
- Shipped **both** templates (PRD hedge said "comparison-explorer only if timeline is brittle" — but timeline came out cleanly in ~200 LOC, so both made it in).
  - `templates/comparison-explorer/` — filterable card grid. Chips per column generate from unique values across rows. Search box queries name + summary + values.
  - `templates/timeline/` — vertical chronological list with sort-direction toggle and tag chips. Lexical date sort handles `YYYY`, `YYYY-MM`, `YYYY-MM-DD` in the same feed.
- Each template is a **real Vite + React + TS project** — package.json, vite.config.ts, tsconfig.json, index.html, src/main.tsx, src/App.tsx, src/index.css, src/data.json, src/data.schema.json, TEMPLATE.md, .gitignore. No monorepo magic.
- Observatory palette baked into `:root` CSS custom properties. No Tailwind, no CSS-in-JS, no component library. Saves install time, version churn, and LLM-authored-component brittleness.
- **Data-driven** by design: only `src/data.json` varies per generation. Template code (App.tsx, CSS) is 100% static. That keeps the LLM's authoring surface small — author structured data, not React components. Massively de-risks the "LLM writes a broken component" failure mode the PRD warned about.
- `TEMPLATE_APP_NAME` placeholder in package.json gets sed'd to the topic slug at scaffold time — avoids pnpm "name collision" warnings if user installs multiple generated apps globally.
- README.md in each generated app documents `pnpm install && pnpm dev` + where to edit data + where to find the TEMPLATE.md hack guide.
- Sidecar (`<slug>-<date>.meta.yaml`) deliberately sits **next to** the app dir, not inside it — so the app dir stays a clean Vite project with no stray YAML.
- **Learnings for future iterations:**
  - The "data is JSON, template is static" split is the right abstraction for LLM-scaffolded apps. When the LLM only has to produce a JSON fixture (not React components), success rate goes from "coin flip" to "near-deterministic". Same principle as quiz's questions.json + static template.
  - Shipping both templates rather than hedging was the right call. Timeline's complexity is not materially higher than comparison-explorer's — both are ~100 LOC of data-driven JSX. Hedging on speculative complexity is a tax.
  - `cp -r` + slug-sed is the whole scaffolder. No Yeoman, no plop, no cookiecutter. The SKILL.md stays legible in one read.
  - Schema files (`data.schema.json`) serve a dual purpose: doc for humans, anchor for future Phase 2E `verify-artifact` re-ingest (it can validate the JSON fixture against the schema before re-rendering cards).
  - Vault-local template override (`.templates/app/<name>/`) was cheap to add — one `[ -d ]` check. Pays for itself the first time a user wants a custom layout for one vault only.
  - `user-invocable: false` pattern across all /generate handlers is now 8-handler consistent. Only the router (/generate) is user-facing. Makes the mental model clean: "one router, N handlers, each a cp-and-fill pipeline over a domain-specific tool".
