What Is Your Job Now?

When AI writes most of the code, your value moves off the keyboard, onto taste, judgment, planning, and review.
"The clankers will write and review the code. Get over it. You will spend your time on planning, validation, taste, judgment, and learning."
Talk · 26 min Source · youtube.com/watch?v=ByOF8qByGHU Captured · 29 Jun 2026 Vault · llm-wiki-ai-research
The argument

The work didn't vanish. It moved.

Thawar frames the whole AI shift through Theory of Constraints, the idea from the novel The Goal: a system has one binding bottleneck at a time, and the moment you remove it, a new one appears. For decades the bottleneck was writing code. AI removed it. So the constraint relocated, upstream to planning and problem framing, and downstream to validation and review.

Then

Your value was the typing

You turned intent into machine code, character by character, in the language and editor you chose.

Now

Your value is the judgment

What to build, why, when, the architecture, understanding the problem, and choosing the green-path solution among the 10,000 possible right ones.

"Where you spend your time is moving. Your value just changes from the person who typed character by character to taste, judgment, and architecture."Farhan Thawar · 2:17

The end of the AI centaur

The "centaur" era (human brain + LLM, pairing to build) ran from 2021 to late 2025. In December 2025, the "Opus 4.5 / GPT 5.2 moment", top engineers concluded the model was simply better at coding than they were. His claim: 90%+ of code should now be AI-written.

The load-bearing ideas

Seven things to internalize

01

The bottleneck always moves

Automate a stage and the constraint jumps elsewhere. Always ask "what's the new bottleneck?" and instrument it before celebrating.

02

Learning is the collateral

The durable asset is what the team learned, not the lines it shipped. Code is now cheap to regenerate; understanding isn't.

03

Code review is the new bottleneck

When AI writes the volume, review decides throughput and quality. Humans were never good at it anyway.

04

Responsibility can't be given away

Use the LLM all you want. Your name goes on the PR. It can be shared, never delegated to a model.

05

A prototype is not production

Don't confuse the map with the territory. Vibe-coded demos are green-path exploration. Delete them; keep the lesson.

06

Scale ambition, don't shrink

Jevons paradox: cheaper code means more ambition per engineer, not fewer engineers. "Why can't 3,000 act like 100,000?"

07

You can't manage by spreadsheet

Lines of code, pull requests per engineer, and token counts are artifacts, not productivity. None of them, even together, tell you if a team is doing the right thing.

Two numbers that make the point

18→3
The Tobi rebuild. 50 engineers, 18 months. CEO asks "if you could start over, how would you build it?" They delete it and rebuild simpler in 3 months. Rule: read the old code, never copy-paste it. The 18 months bought the learning that made the 3-month version possible.
$600K
The intern's six lines. One intern deleted 6 lines of code and saved over $600,000 a year in infrastructure. No output metric on any dashboard would have flagged it. Outcomes over output.

The operating manual

The five ways Shopify works

  1. Model selection: only the biggest models

    Engineers may only use the largest, most expensive models. Human time is worth more than clanker time: a small model's subtle bug costs hours of human debugging that a big model avoids.

  2. Code review is the bottleneck → a Council of LLMs

    "The four scariest letters: LGTM." Humans weren't reading the code. Review it with a panel of models, each on a different task and lens, deciding ship-or-send-back. Better than humans, not perfect.

  3. Responsibility cannot be given away

    Have the LLM write everything, do your validation, but your name is on the PR. Tag the model if you like. It's about you, not the model.

  4. Hire with a 3-tier test

    No-AI (solve by hand, prove the fundamentals), AI-optional (pull it in if you want), AI-mandatory (a problem so large you must wield the tool). Across all three you get a real signal.

  5. Scale ambition, not headcount

    "Can I make my dev team smaller?" is the unambitious question. The play is more entrepreneurship and more productivity per person.

The pattern worth stealing

The Council of LLMs

The single most transferable idea in the talk: don't review a change with one reviewer. Use a panel of models, each given a different role, a different lens, and a different reasoning budget, composed into one gate that decides ship or send-back-with-commentary.

The differentiator is the lenses, not the model count. A security lens, an accessibility lens, a maintenance lens, and an architecture lens each see a different slice of the same diff, and catch failure modes that one reviewer never would.

So what do you actually do

Action points

Concrete moves, grouped by who owns them. Tick them off.

For you, the individual engineer
Shrink your time-to-reach-for-AI. Measure reflexivity by reaction time: when you hit a problem, how fast do you pull AI in? Make that distance small.
Default to the biggest model. Your time is the expensive resource. Stop economizing on tokens to save a subtle-bug debugging session.
Spend the freed time upstream and downstream. Plan harder, frame the problem with the customer, and validate output. That's where the bottleneck is now.
Put your name on the PR and own it. Tag the model if you want, but responsibility is yours. Validate before you ship.
Answer "what did I learn this week?" Not "what did I ship." Name one secret you now believe that others don't, because you tried it.
For the team / engineering lead
Make code review the most-engineered stage. It's the new bottleneck. Stand up a multi-model review gate before optimizing anything else.
Build a Council of LLMs with real lenses. Security, accessibility, maintenance, architecture, each a distinct reviewer, composed into one ship/send-back verdict.
Delete the prototype, keep the learning. Read the old code, never copy-paste it. Rebuild for production. Resist the map-for-territory trap.
Stop managing by spreadsheet. Retire the token leaderboard for a value dashboard. Use weekly demos to gauge velocity, not LoC or PR counts.
Set circuit breakers, not token limits. Don't cap exploration. Surface runaway spend and let a human say keep-going or stop.
Raise ambition, don't cut headcount. Ask "how do our N engineers act like 10N?" Pull the roadmap in: why not this month, this week, today?

From talk to build

What this means for the agent fleet

This talk is the external validation for building a Council of LLMs into your own tooling: a fleet of specialized review agents, each with one lens, gating every push, with humans keeping responsibility and taste.

The fleet, mapped to lenses

Each agent is one role on the council. Different focus, different docs linked, composed into a pre-push gate that returns severity-tagged findings and routes failures back to the implementer.

AgentLensReads from
code-reviewercorrectness, the general gatecode-review skill
architectureboundaries, coupling, deep modulescoding-principles
simplicity (YAGNI / rule-of-3)over-engineering, premature abstractionsimplicity, abstraction
solid-drySRP, DRY, single source of truthsolid, abstraction
securityauthz, secrets, attack surfacesecurity-audit
way-of-workingtests, commits, definition-of-doneway-of-working
dev (implementer)fixes what the council flagsthe report

The loop

A pre-push hook runs the council, writes an "investigate" report, and blocks the push on critical findings. The report routes back to the agent that wrote the code if it's still alive, or spins up a fresh dev agent to work it. A per-project config file (with a first-run prompt) decides whether the review runs at all, so it stays opt-in per repo.
